# Task brief template

Adapt freely — a few honest sentences beat filled-in boilerplate. The
sections cover what Claude plans best with: intent, grounding, verifiable
success, and limits. Delete any that don't apply.

**Task** — what to build or fix, and the problem it solves. Stating the
motivation, not just the change, lets Claude resolve small ambiguities the
way you would.

**Grounding** — where to start: files, directories, commands, logs, or
decisions already made.

**Success** — how the result is judged: behavior you can observe, tests or
commands that must pass, output you expect.

**Out of scope** — what must stay untouched: interfaces, dependencies,
files, or behavior to preserve.

Work it end to end. Decide reversible details yourself; raise a choice
before acting only when it would change something stated above.
